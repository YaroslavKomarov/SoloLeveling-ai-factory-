> Дата: 2026-03-07 | Ревьюер: Claude Code

Три проверяемые механики:
1. **[CAL]** Проверка доступного времени в календаре при создании цели
2. **[RETRO]** Сценарий запуска и проведения ретроспектив
3. **[REDIST]** Алгоритм перераспределения стратегических задач

## Что реализовано по календарю (важный контекст)

Перед списком багов — важно понимать, что **частично** уже есть:

**После подтверждения цели** (`/api/goals/confirm`, строки 172–291):
- Запрашиваются занятые интервалы из Google Calendar per-day (`fetchBusyIntervals`)
- Вычисляются свободные слоты внутри окна активности пользователя (`findFreeIntervals`)
- Задачи получают время начала внутри свободных слотов (`computeTaskStartTimes`)
- Создаются события в Google Calendar (`createTaskEvent`)

**Что НЕ реализовано** — проверка **до** показа превью плана:
- Агент генерирует даты задач без учёта занятости конкретных дней
- Нет проверки, помещается ли вся цель в доступное время за 90 дней
- Нет упрощения или блокировки

## Критические баги

### BUG-01 [CAL] [CRITICAL] — Нет предварительной проверки feasibility: цель планируется без учёта реального времени в календаре

**Затронутые файлы:**
- `src/app/api/agents/goal-generator/route.ts` — агент не получает данные о занятости календаря
- `src/lib/agents/goal-generator/prompt.ts` — только флаг `calendarConnected`, не реальные слоты
- `src/lib/agents/goal-generator/tools.ts` — нет инструмента для calendar feasibility
- `src/lib/calendar/slot-finder.ts` — функции есть, но вызываются только после подтверждения

**Описание (детально):**

Цепочка сейчас:
```
Агент генерирует план (даты задач)
→ Пользователь видит превью
→ Подтверждает
→ ТОЛЬКО ТОГДА запрашивается календарь (post-confirm, нет пути назад)
```

Что должно быть:
```
Агент генерирует черновик плана (количество задач, длительности)
→ Система запрашивает FreeBusy за 90 дней
→ Считает: totalFreeMinutes vs totalTaskMinutes
→ Если хватает → распределяет задачи по свободным дням, показывает превью
→ Если не хватает → упрощение или блокировка (см. BUG-02, BUG-03)
```

**Дополнительная проблема с текущим post-confirm sync:**
Если в конкретный запланированный день нет свободного слота под задачу, `computeTaskStartTimes` возвращает `null`, и задача создаётся с `calendar_event_id` = null — т.е. событие в календаре не создаётся, задача в плане **не видна** пользователю в Google Calendar. Никакого предупреждения нет (confirm/route.ts:252-275 — fallback `startTimes[i] ?? activityStart` создаёт событие поверх занятого слота).

**Влияние:**
1. Пользователь может создать цель с 200 задачами при реально доступных 50 слотах за 90 дней
2. События создаются поверх существующих встреч (конфликты в Google Calendar)
3. Нет никакой обратной связи о реальной вместимости расписания

### BUG-02 [CAL] [CRITICAL] — Нет алгоритма упрощения цели на 30–40%

**Затронутые файлы:**
- `src/lib/agents/goal-generator/tools.ts`
- `src/lib/agents/goal-generator/prompt.ts`
- `src/components/goals/GoalCreationDialog.tsx`

**Описание:**

Когда системная проверка (BUG-01) выявит нехватку времени, должна сработать логика упрощения:

1. Посчитать дефицит: `deficit = totalTaskMin - availableMin`
2. Если `deficit / totalTaskMin ≤ 0.40`: упростить на нужный процент
   - Уменьшить `strategicTaskCount` и `regularTaskCount` пропорционально по квестам
   - Показать пользователю упрощённый вариант с объяснением (`"Упрощено на X%, чтобы вписаться в ваше расписание"`)
   - Получить аппрув → сохранить упрощённый план
3. Если `deficit / totalTaskMin > 0.40`: см. BUG-03 (блокировка)

Ничего из этого не существует. Нет ни инструмента в `goalGeneratorTools`, ни логики в UI, ни API endpoint для передачи данных о доступном времени обратно в агент.

**Влияние:** Пользователь не получает адаптированного плана — либо берётся перегруженный план, либо нет подсказок.

### BUG-03 [CAL] [CRITICAL] — Нет блокировки создания цели при критической нехватке времени

**Затронутые файлы:**
- `src/components/goals/GoalCreationDialog.tsx`
- `src/app/api/goals/confirm/route.ts`

**Описание:**

Если даже упрощение на 40% не даёт достаточно времени, создание цели должно блокироваться с сообщением:
> "Недостаточно свободного времени в течение 90 дней. Отмените другие цели или освободите место в календаре."

Сейчас `confirm/route.ts` проверяет только:

- обязательные поля

Никакой проверки доступного времени нет. Цель создаётся в любом случае.

**Влияние:** Пользователь создаёт нереализуемые цели, которые незаметно превращаются в calendar-spam (события поверх встреч) или задачи без событий.

### BUG-04 [CAL] [CRITICAL] — Нет учёта типа активности при размещении задач в слоты

**Затронутые файлы:**
- `src/lib/calendar/slot-finder.ts` — `computeTaskStartTimes` (строки 125-193)
- `src/app/api/goals/confirm/route.ts` (строки 240-243)

**Описание:**

По ТЗ задачи должны распределяться с учётом типа активности:
- **утро** → интеллектуальные (`fatigue_type: 'intellectual'`)
- **обед/день** → физические (`fatigue_type: 'physical'`)
- **вечер** → эмоциональные (`fatigue_type: 'emotional'`)

Текущая реализация `computeTaskStartTimes` принимает только `{ duration_minutes }` и раскидывает задачи **равномерно** по всем свободным слотам без учёта времени суток или типа задачи.

В `confirm/route.ts:240-243` задачи передаются в slot-finder без `fatigue_type`:
```typescript
const taskInputs = dateTasks.map((t) => ({
  duration_minutes: t.duration_minutes ?? (t.task_type === 'strategic' ? 27 : 12),
}))
```

**Требуемый алгоритм (per-day):**
```
Для каждого дня:
  1. Разбить окно активности на периоды: утро / день / вечер
  2. Интеллектуальные задачи → предпочтительно утренние слоты
  3. Физические → дневные слоты
  4. Эмоциональные → вечерние слоты
  5. Если предпочтительный период занят → fallback в любой свободный слот
```

**Влияние:** Все задачи размещаются в рандомном порядке в течение дня. Умной оптимизации по типу активности нет.

### BUG-05 [CAL] [HIGH] — Timezone bug в `fetchBusyIntervals`: UTC-время сравнивается с локальным activity window

**Затронутые файлы:**
- `src/lib/calendar/slot-finder.ts:58-69`

**Описание:**

Google FreeBusy API возвращает интервалы в **UTC**. Код извлекает время строчным срезом:
```typescript
const s = b.start.substring(11, 16) // "06:00" для UTC
const [sh, sm] = s.split(':').map(Number)
return { startMin: sh * 60 + sm, ... }
```

Activity window пользователя (`activity_window_start: "09:00:00"`) хранится в **локальном времени**.

Для пользователя UTC+3 с activity window 09:00–21:00:
- Реальная встреча 09:00 локального → `start` в FreeBusy = `"2026-03-07T06:00:00Z"`
- Код извлекает `"06:00"` → `startMin = 360`
- Но window start = `540` (09:00)
- Встреча **вне окна активности** → считается свободным временем → задача ставится поверх встречи

**Влияние:** Для пользователей не в UTC (а это большинство) система игнорирует встречи, которые находятся в пределах рабочего дня, и создаёт задачи поверх них. Степень ошибки = UTC offset пользователя.

**Примечание:** Это баг в уже работающем коде post-confirm sync, не в несуществующем pre-confirm check.

### BUG-14 [RETRO] [MEDIUM] — Промпт агента ретроспективы не запрещает произвольное изменение содержания задач

**Затронутые файлы:**
- `src/lib/agents/retrospective-analyzer/prompt.ts:33-40`

**Описание:**

Тип корректировки `task_content` описан как:
> "Modify task title or description to better match user's actual capacity"

Это слишком широко. Агент может переписать содержание задачи произвольно — убрать смысловое ядро, изменить цель задачи, сделать задачу принципиально другой. Согласно ТЗ, переформулировка должна **только** смещать фокус задачи (какой аспект практиковать), не меняя структуру плана, сроки и общую направленность.

**Пример правильной переформулировки:**
```
До: "Тренировка на выносливость (30 мин)"
После: "Тренировка верхней части тела (30 мин)"
Причина: пользователь сообщил, что пренебрегает верхним телом
```

**Пример запрещённой переформулировки:**
```
До: "Написать технический дизайн модуля X"
После: "Изучить документацию по модулю X"
Причина: убрана ключевая активность создания артефакта, задача потеряла смысл в контексте цели
```

**Что отсутствует в промпте:**
1. Явный запрет изменять даты, сроки, количество задач или квесты (структуру плана)
2. Явное указание: изменения должны быть обоснованы конкретной обратной связью пользователя (что именно лагает)
3. Ограничение: переформулировка — это смена _фокуса/акцента_ задачи, а не смена её природы

**Влияние:** Агент может создавать корректировки, которые превращают достижимую цель в недостижимую или меняют смысл задачи в рамках квеста без обоснования пользовательским фидбеком.

### BUG-06 [RETRO] [HIGH] — Агент-анализатор ретроспективы не запускается, когда нет активных целей со статистикой

**Затронутые файлы:**
- `src/components/retrospective/RetrospectiveWizard.tsx:68-119`

**Описание:**
Вызов агента-анализатора (`/api/agents/retrospective-analyzer`) происходит **только** в функции `handleNextFromGoal()`, которая вызывается при переходе с последней страницы обратной связи по цели. Если у пользователя нет активных целей со статистикой за неделю (`goalsWithStats.length === 0`), страниц обратной связи не создаётся. Страница статистики (page 0) переходит напрямую на страницу корректировок (page 1) через `nextPage()` — без вызова `handleNextFromGoal()`, то есть без запуска агента.

**Воспроизведение:** Ретроспектива создаётся в первую неделю, когда у пользователя ещё нет завершённых или пропущенных задач. `goalsWithStats` пустой → агент не запускается → adjustments пустые.

**Влияние:** Страница корректировок показывает "No adjustments proposed. You are on track." без реального анализа.

### BUG-07 [RETRO] [HIGH] — Завершение ретроспективы падает с 409, если не было страниц обратной связи

**Затронутые файлы:**
- `src/app/api/retrospectives/[retroId]/complete/route.ts:56`
- `src/app/api/retrospectives/[retroId]/feedback/route.ts:77-79`

**Описание:**
Ретроспектива переходит из статуса `pending` в `in_progress` только когда отправляется первый feedback (route.ts:77-79). Если `goalsWithStats.length === 0` (см. Когда пользователь нажимает "Complete Retrospective", endpoint проверяет `retro.status !== 'in_progress'` и возвращает **409 Conflict**.

```typescript

  return NextResponse.json({ error: `Retrospective must be in_progress...` }, { status: 409 })
}
```

**Воспроизведение:** Первая ретроспектива пользователя без задач за неделю → нажать "Complete Retrospective" → 409 ошибка в консоли, wizard не закрывается.

**Влияние:** Пользователь не может завершить ретроспективу, wizard заблокирован.

### BUG-13 [UI] [MEDIUM] — Устаревший контент баннеров на странице цели

**Затронутые файлы:**
- `src/components/goals/GoalAtRiskBanner.tsx`
- `src/app/app/goals/[goalId]/GoalDetailClient.tsx:408-449`

**Описание:**

**1. GoalAtRiskBanner — текст от удалённого redistributor**

Баннер рендерится при `goal.is_at_risk && goal.status === 'active'` (строка 408). Текст внутри компонента говорит о стратегических задачах и дедлайне — это концепция redistributor, который удалён. Теперь `is_at_risk = true` означает "2 из 7 пропусков регулярной задачи".

Должно быть:
> "Предупреждение: '[taskTitle]' пропущена 2 из 7 раз. Следующий пропуск провалит цель."

`taskTitle` нужно вычислить в `GoalDetailClient` из `allTasks` — найти группу siblings (same `title + goal_id`) с количеством пропущенных/skipped = 2 и передать как проп.

**2. Failed notice — хардкод устаревших failure_reason строк**

Строки 443–446 `GoalDetailClient.tsx`:
```typescript
goal.failure_reason === 'consecutive_skips'
  ? 'Three consecutive sessions were missed without completion.'
  : 'The overall task skip rate exceeded 20%.'
```

Новые форматы `failure_reason`:
- `skip_threshold:<taskTitle>` — задача пропущена ≥3 из 7
- `quest_failure:<questTitle>` — квест закрыт менее чем на 70% XP (проверяется в день дедлайна)

Ни один из старых хардкодов не совпадёт → всегда показывается "The overall task skip rate exceeded 20%." вне зависимости от реальной причины.

Формат `failure_reason` для квестового провала: `quest_failure:<questTitle>:<completionPct>` (процент хранится в строке, чтобы не нужно было пересчитывать).

Должно быть:
```
skip_threshold:<taskTitle>
  → "Цель провалена: задача '<taskTitle>' пропущена 3 или более раз из 7"

quest_failure:<questTitle>:<pct>
  → "Цель провалена: квест '<questTitle>' выполнен на <pct>% (необходимо 70%+)"

иначе → "Цель провалена"
```

**Влияние:** Пользователь видит неверный текст в обоих баннерах. `GoalAtRiskBanner` говорит о несуществующей проблеме; `failed notice` всегда показывает неверную причину провала.

### BUG-17 [FAILURE] [HIGH] — Проверка провала квеста по дедлайну не реализована

**Затронутые файлы:**
- `supabase/functions/nightly-planning/index.ts` — нет шага проверки end_date

**Описание:**

По бизнес-логике цель может провалиться в день дедлайна, если любой квест набрал менее 70% XP (сумма XP выполненных задач квеста / целевой XP квеста). Это второй сценарий провала, отличный от `skip_threshold`.

Сейчас ночная функция не содержит шага, который на дату `goal.end_date` проверяет completion rate каждого квеста. Если пользователь пропускал стратегические задачи (не вызывающие skip_threshold) и часть регулярных (≤2 из 7, не триггеря провал), квест может остаться незакрытым на 70%, но цель продолжит жить как active.

**Требуемое поведение (в ночной функции):**
```
Если today == goal.end_date:
  Для каждого квеста цели:
    completionXp = SUM(xp_reward) WHERE task.quest_id = quest.id AND status = 'completed'
    targetXp    = SUM(xp_reward) WHERE task.quest_id = quest.id
    completionPct = Math.round(completionXp / targetXp * 100)
    if completionPct < 70:
      failGoal(goalId, `quest_failure:${quest.title}:${completionPct}`)
      отправить push — "Цель '...' провалена: квест '${quest.title}' выполнен на ${completionPct}% (нужно 70%+)"
      break
  Если все квесты >= 70%:
    goal.status = 'completed'
    отправить push — "Цель '...' достигнута! Итого XP: ..., квестов закрыто: N/N"
```

**Нет реализации:**
1. Сам шаг проверки `end_date` в ночной функции
2. Push-уведомление о результате цели (успех / провал с причиной)
3. Нет перевода цели в `status = 'completed'` по дедлайну (только `failed` есть)

**Влияние:** Цели после дедлайна висят как `active` бесконечно. Пользователь не получает итоговое уведомление о результате.

## Статус реализации механик

### 1. Календарь при создании цели

**Что есть:** post-confirm синхронизация работает — задачи размещаются в свободные слоты Google Calendar (slot-finder.ts), создаются события. Но только после того, как пользователь уже подтвердил план.

**Чего нет:**
- Pre-confirm feasibility check (BUG-01) — отдельный спринт
- Упрощение цели (BUG-02) — отдельный спринт
- Блокировка (BUG-03) — отдельный спринт
- Умное планирование по типу активности (BUG-04) — реализуется модификацией `computeTaskStartTimes`
- Timezone fix (BUG-05) — быстрый fix

### 2. Ретроспективы
**⚠️ Базовый happy path работает.** Блокирующие edge cases: BUG-06 + BUG-07 (первая ретроспектива без задач), BUG-08 (race condition). Промпт агента требует уточнения ограничений переформулировки (BUG-14).

### 3. Пропущенные стратегические задачи
**✅ Новый дизайн:** пропущенные задачи (detected overnight) показываются в отдельном разделе страницы TODAY. Redistributor и compaction удалены.

### 4. Логика провала цели
**❌ Не соответствует OKR-модели.** Текущий механизм — per-task skip rate (3 подряд или 20%), а должен быть per-quest (≥70% = успех). Стратегические задачи вообще исключены из проверки провала (BUG-16).
