# SoloLeveling v2 - Документ для инициализации проекта

## Краткое описание

**Название:** SoloLeveling v2
**Тип:** PWA приложение для планирования и достижения целей
**Методология:** ASE v3.0 (Adaptive Strategic Execution) - адаптивная стратегическая реализация
**Архитектура:** LLM-First / Агентная (LangGraph.js)
**Стек:** Next.js 14+, TypeScript, Supabase (PostgreSQL + pgvector), OpenRouter (Claude 3.5 Sonnet)
**Дизайн:** Темная готика, минимализм (детали в `./design/`)

---

## 1. Методология ASE v3.0

### 1.1 Иерархия целей

**Сферы (Spheres):**
- Это жизненные домены: Работа, Здоровье, Обучение, Отношения и т.д.
- Просто категории для группировки целей, без особой логики

**Цели (Goals):**
- Конкретные задачи на 3 месяца (всегда 90 дней)
- Основаны на OKR методологии (Objective + Key Results)
- Два типа целей: skill-based (практика) и knowledge-based (теория)

**Квесты (Quests / Key Results):**
- 3-5 измеримых результатов на каждую цель
- Примеры: "Пройти 20 задач по алгоритмам", "Прочитать 5 глав книги"
- Имеют числовые значения (target, current)

**Задания (Tasks):**
- Атомарные действия на 10-30 минут
- Дают XP за выполнение (геймификация)
 - Два типа: регулярные и стратегические

### 1.2 Типы целей

**Skill-based (практико-ориентированные):**
- Фокус на отработку навыков через повторение
- Соотношение регулярных и стратегических задач гибкое, система решает сама, но в пользу регулярных
- Пример: "Научиться играть на гитаре"

**Knowledge-based (теоретико-ориентированные):**
- Фокус на изучение теории и концепций
- Соотношение регулярных и стратегических задач гибкое, система решает сама, но в пользу стратегических
- Пример: "Изучить квантовую физику"

### 1.3 Типы заданий

**Регулярные задания (Regular Tasks):**
- Длительность: 12 минут (`duration_minutes = 12`)
- Награда: 50 XP
- Используют интервальное повторение (алгоритм Эббингауза: 1, 2, 4, 7, 14, 30, 60 дней)
- Нельзя переносить между днями (жесткий график повторений)
- Система планирует их на конкретное рекомендованное время (time-boxing) в календаре, но пользователь может выполнять в любое время дня по своему усмотрению
- Регулярные задания не перепланируются — всегда идут по своему интервальному графику, несмотря на пропуски

**Стратегические задания (Strategic Tasks):**
- Длительность: 27 минут (`duration_minutes = 27`)
- Награда: 100 XP
- Система также планирует их на конкретное рекомендованное время (time-boxing) в календаре, но пользователь может выполнять в любое время дня. Главное — в течение дня и до запуска ночной логики планирования, чтобы задание не было засчитано как пропущенное
- Можно перераспределять между днями при необходимости: если пользователь пропускает стратегическое задание, то система при следующем ночном планировании запускает алгоритм перераспределения нагрузки и пытается вместить пропущенные задания в оставшийся план
- Для завершения стратегического задания обязательно заметка-отчёт минимум 50 символов И 8 слов (валидируется и на клиенте, и на сервере)

**Общее для всех заданий:**
- Все задания планируются в свободные слоты периода активности пользователя (на основе его профиля и паттернов)
- Задания нельзя "провалить" — их можно либо выполнить, либо не выполнить (также можно начать и отменить, что равнозначно невыполнению)
- Нельзя выполнять задания из прошлых/будущих дней — только сегодняшние
- Для старта задания: пользователь нажимает кнопку "Start" → запускается таймер обратного отсчёта. Одновременно может быть активно только одно задание
- Активное задание отображается в виде плавающего баннера (ActiveTaskBanner) вверху всех страниц с обратным отсчётом и кнопкой отмены
- При достижении 0: задание автозавершается (POST /api/tasks/[id]/complete)
- Состояние таймера сохраняется в localStorage (`sl_active_task`); при возврате в приложение — таймер восстанавливается; если время истекло в офлайне — задание автозавершается при следующем открытии

**Условия провала цели:**
- Если одно и то же регулярное задание пропущено 3 раза подряд → цель провалена
- Если суммарный процент пропусков регулярных заданий по цели достигает 20% → цель провалена
- При провале: прогресс по стратегическим заданиям и заметки к ним сохраняются, прогресс по регулярным заданиям сбрасывается. Система предложит сформулировать новую цель на основе проваленной (уже с учетом полученных знаний от стратегических задач)

### 1.4 Система усталости (Fatigue)

**Три типа усталости:**
- Физическая (Physical): 0-100%
- Эмоциональная (Emotional): 0-100%
- Интеллектуальная (Intellectual): 0-100%

**Правила:**
- Каждое задание имеет стоимость по каждому типу усталости (4% на регулярные и 6% на стратегические)
- Формального лимита нет — усталость может д��стигать 100%, но система предупреждает и рекомендует отдохнуть (не блокирует выполнение заданий)
- Сброс: каждый день в 00:00 все типы обнуляются
- Визуализация: всегда видна в верхней панели пользователя (три стамины)

**Цветовое кодирование усталости:**
- 0-70%: белый цвет
- 71-85%: желтое свечение
- 86-95%: оранжевое свечение
- 96-100%: красное свечение (с иконкой предупреждения и рекомендацией о��дохнуть)

### 1.5 Система опыта и уровней

**Получение опыта (XP):**
- Регулярное задание: 50 XP
- Стратегическое задание: 100 XP

**Формула для повышения уровня:**
- Требуемый XP для следующего уровня: `100 * уровень^1.5`
- Пример: для 2 уровня нужно 282 XP, для 3 уровня - 519 XP

**Повышение уровня:**
- При достижении порога срабатывает анимация level-up
- Полноэкранный модал в стиле визуальной новеллы
- Показывается новый уровень, общий XP, требование до следующего уровня

### 1.6 Геймификация

**Элементы:**
- Прогресс-бары для XP, квестов, усталости
- Счетчики уровня и опыта
- Дерево навыков (skill tree): горизонтальные ветки по сферам. Каждая сфера = строка с узлами-целями слева направо (от старых к новым). Активная цель подсвечена, завершённые — заполненные/приглушённые, провалившиеся/отменённые — серые. Нет переключателя вид/список — только дерево
- Анимации при выполнении заданий
- Экраны уведомлений при достижении целей (модалки)

## 2. Техническая архитектура

### 2.1 Технологический стек

**Frontend:**
- Next.js 14+ с App Router и Server Components
- React 18+
- TypeScript (строгий режим)
- Tailwind CSS (кастомная конфигурация под дизайн-систему)
- Zustand для глобального состояния
- React Hook Form + Zod для форм и валидации
- Recharts для графиков
- Lucide React для иконок

**Backend:**
- Supabase:
  - PostgreSQL для основной БД (включая контент заметок как `TEXT`)
  - pgvector для эмбеддингов (RAG система)
  - Supabase Auth для авторизации
  - Supabase Storage только для изображений (`![[image.png]]`)
  - Supabase Realtime для подписок на изменения
- Anthropic SDK напрямую (без OpenRouter):
  - `claude-haiku-4-5-20251001` — daily-planner, task-redistributor, knowledge-rag
  - `claude-sonnet-4-6` — goal-generator, retrospective-analyzer, goal-dialog-agent
- Vercel AI SDK (`streamText`, `generateObject`, tool use) для оркестрации агентов

**Интеграции:**
- Google Calendar API (OAuth 2.0, webhooks) — обязательная интеграция
- Obsidian-совместимый формат для базы знаний
- Telegram Bot API для push-уведомлений

**Деплой:**
- Vercel для Next.js приложения
- Supabase Cloud для БД
- PWA с service workers для оффлайн-поддержки

### 2.2 Агентная архитектура (Vercel AI SDK)
ВАЖНО: является приблизительной и может корректироваться в процессе воплощения

**Основные агенты:**

**1. Агент генерации целей (goal-generator):**
- Получает: профиль пользователя из @me, сф��ру, описание цели и контекст диалога с пользователем
- Анализирует: достижимость цели с учетом расписания в календаре и других активных целей
- Генерирует: 3-5 квестов (где каждый квест - ключевой результат по цели) и 3-месячный план заданий
- Выдает: объект цели с квестами и заданиями на каждый день
- Инструменты: расчет усталости, проверка конфликтов календаря, валидация нагрузки

**2. Агент ежедневного планирования (daily-planner):**
- Получает: активные цели, текущую дату, состояние ус��алости
- Генерирует: список заданий на сегодня с учетом интервального повторения и чередования нагрузки
- Проверяет: календарь, чтобы не возникло конфликтов, и задания планирует в свободные слоты периода активности пользователя
- Определяет: какие задания были пропущены за прошлый день и запускает соответствующую логику (перераспределение для стратегических, увеличение счетчика пропусков для регулярных)
- Выдает: список заданий с расчетом стоимости усталости
- Инструменты: получение событий календаря, расчет дневной усталости, применение интервального повторения, чередование заданий по типу усталости и по целям (interleaving), расстановка перерывов между заданиями (правила в SC-36)

**3. Агент перераспределения (task-redistributor):**
- Получает: невыполненные стратегические задания, оставшееся время цели
- Алгоритм: равномерное распределение по оставшимся дням → если не помещается → критический алерт
- Выдает: обновленное распределение заданий или критический алерт
- Инструменты: расчет оставшейся емкости, проверка достижимости

**4. Агент ретроспективы (retrospective-analyzer):**
- Получает: прогресс целей, статистику выполнения, обратную связь пользователя
- Анализирует: паттерны поведения, успешность выполнения
- Предлагает: корректировки количества накапливаемой усталости за задания (можно только уменьшать)
- Выдает: инсайты, обновленный план, паттерны для @me/patterns.md
- Инструменты: анализ процента выполнения, предложение корректировок, обновление паттернов

**5. Агент базы знаний RAG (knowledge-rag):**
- Получает: запрос пользователя, контекст текущей заметки
- Процесс: семантический поиск через pgvector, обход wikilinks, сборка контекста
- Выдает: ответ со ссылками на источники
- Инструменты: поиск по эмбеддингам, обход wikilinks, рендеринг markdown

**6. Агент диалога по цели (goal-dialog-agent):**
- Роль: мыслитель, наставник, гуру цели — творческий помощник и эксперт по предметной области цели
- Получает: текущий прогресс по цели, квесты, общий план, связанные заметки из KB, постановку текущего стратегического задания
- Управление контекстом: саммари предыдущих обсуждений + последние N сообщений
- Выдает: помощь в решении задания, синтез выводов, создание заметок
- Основной use case: среда выполнения стратегических заданий

**Структура состояния агентов:**
- Пользователь: ID, профиль из @me файлов, уровень, XP
- Цели: активные, достигнутые, проваленные, отмененные
- Задания на день
- Текущая усталость (3 типа)
- События календаря
- Контекст для RAG

**Коммуникация между агентами:**
- Агенты реализованы как Next.js API routes с Vercel AI SDK
- Промпты хранятся в директории prompts (версионируются)
- Схемы инструментов определены в директории tools (Vercel AI SDK tool use)

### 2.3 База данных (Supabase PostgreSQL)
ВАЖНО: структура примерная и может поддаваться корректировкам

**Основные сущности системы:**
- Пользователи (профиль, уровень, XP, привязка к календарю)
- Сферы жизни (категории целей)
- Цели (тип, статус, даты, оценка достижимости, привязка к сфере)
- Квесты / Key Results (измеримые результаты, привязка к цели)
- Задания (тип, статус, дата, усталость, повторения, привязка к квесту)
- Ежедневная усталость (три типа на каждый день)
- Заметки (метаданные + контент как `TEXT` в PostgreSQL; изображения в Storage)
- Эмбеддинги для RAG (pgvector)
- Паттерны поведения (автообновление агентом)
- Ретроспективы (обратная связь, анализ, изменения плана)
- Кэш событий Google Calendar

Конкретная структура таблиц, связи и индексы определяются на этапе реализации с учетом актуальных требований.

### 2.4 База знаний (Obsidian-совместимый формат)

**Хранение:**
- Контент заметок: PostgreSQL `notes.content TEXT` (не Storage)
- Изображения: Supabase Storage (только бинарные файлы, `![[image.png]]`)
- Структура файлов как в Obsidian vault:
  - `@me/` - 6 основных файлов профиля + patterns.md (автоматический)
  - `{имя-сферы}/sphere.md`
  - `{имя-сферы}/{имя-цели}/goal.md`
  - `{имя-сферы}/{имя-цели}/{имя-заметки}.md`

**Формат markdown:**
- Frontmatter (YAML) в начале файла: метаданные (название, теги, даты)
- Основной контент: markdown с wikilinks `[[название-заметки]]` и тегами `#тег`
- Вставка изображений: `![[image.png]]`

**Процесс синхронизации:**
- Пользователь редактирует заметку → автосохранение в PostgreSQL (debounce 1.5s)
- Парсер извлекает frontmatter, wikilinks, теги → обновляет метаданные
- Контент делится на чанки → генерируются эмбеддинги для RAG
- Изменения wikilinks → пересчет графа связей

**UI (три панели):**
- **Левая:** Дерево директорий и файлов, поиск файлов по названию, список тегов, фильтры
- **Центральная:** Отрендеренный markdown, backlinks внизу, кнопка редактирования
- **Правая:** RAG чат с контекстом из текущей заметки + связанных заметок (2 уровня вглубь)

## 3. Полное описание потоков (7 flows, 38 сценариев)

### Flow 1: Инициализация и настройка профиля

**SC-01: Онбординг пользователя**

При первом запуске приложения:
- Приветственный экран в стилистике проекта (темная готика)
- Кнопка "Начать" → регистрация через email/пароль или Google auth (Supabase Auth)
- После регистрации автоматический онбординг:
  1. Диалог с системой: краткий рассказ о системе и получение данных пользователя, необходимых для создания структуры @me (6 файлов: profile, career, skills, interests, personality, settings)
  2. Инициализация profile.md с минимальными данными (имя, часовой пояс, период активности для планирования целей). Часовой пояс и период активности можно будет изменить в настройках
  3. Подключение Google Calendar (обязательный шаг, нельзя пропустить) → OAuth flow → сохранение токена → настройка синхронизации
  4. Установка пользователем времени проведения ретроспектив (напоминание раз в неделю), можно потом изменить в настройках
  5. Переход на экран "Создайте первую сферу"

**SC-02: Структура профиля @me**

Плоская директория с 6 markdown файлами:
- `profile.md`: имя, возраст, профессия, часовой пояс, рабочий период
- `career.md`: карьерные цели, текущая роль, амбиции
- `skills.md`: текущие навыки, уровни, области для развития
- `interests.md`: хобби, интересы, мотивации
- `personality.md`: стиль работы, предпочтения, паттерны энергии

Дополнительно:
- `patterns.md`: автоматически создается и обновляется агентом ретроспективы
- Пользователь может просматривать, но не редактировать

Формат:
- Frontmatter (YAML) для структурированных данных
- Основная часть - свободный текст в markdown
- Все файлы редактируются через UI базы знаний

**SC-03: Подключение Google Calendar**

Подключение обязательно при онбординге. На странице настроек доступно управление:
- Кнопка "Подключить календарь" → OAuth 2.0 flow с Google
- Разрешения: `calendar.events` scope — чтение и управление событиями (создание, удаление)
- После авторизации: сохранение зашифрованного токена (AES-256-GCM), настройка синхронизации событий
- Тест подключения: загрузка событий на сегодня, показ количества
- Кнопка "Отключить календарь" (удаляет токен)
- Без подключенного календаря система не может планировать задания в свободные слоты
- Существующие пользователи, подключившие календарь со старым скоупом `calendar.readonly`, должны переподключить для получения прав записи

**SC-04: Расписание ретроспектив**

Во время онбординга:
- Агент предлагает еженедельную ретроспективу (например, воскресенье 18:00)
- Пользователь может настроить день и время, указав свое
- В назначенное время отправляется push-уведомление в Telegram (и вообще важные уведомления надо туда слать)
- Прошлые ретроспективы можно просмотреть на странице История в любое время

### Flow 2: Генерация и валидация целей

**SC-05: Создание сферы**

- Пользователь создает сферу: название (обязательно), описание (опционально), иконка (lucide icon, никаких эмодзи никогда)
- Без валидации (это просто категория)
- Создается файл `{имя-сферы}/sphere.md` в базе знаний

**SC-06: Ввод описания цели**

- Пользователь выбирает сферу → кнопка "Создать цель"
- Диалоговый формат с системой для постановки цели по OKR: система задает вопросы, чтобы максимально подробно составить план на 3 месяца для достижения цели. Система может задавать уточняющие вопросы до тех пор, пока не будет достаточной ясности о цели и ее критериях
- Один сквозной чат на весь процесс создания цели. Сообщения сохраняются временно — если диалог начат, но не завершен, пользователь может вернуться и продолжить. После окончательного создания цели история диалога удаляется
- Управление контекстным окном: если диалог затягивается, система формирует саммари предыдущих сообщений (аналогично SC-19), чтобы не переполнить контекст LLM
- Система сама по контексту определяет тип цели: skill-based (цели, которые строятся вокруг развития навыка: мало стратегических задач и много регулярных) или knowledge-based (цели, которые строятся на приобретении знаний: много стратегических задач и меньше регулярных)
- Система сама составляет детальный план: сначала бьет на квесты (где каждый квест — ключевой результат), затем декомпозирует квесты на стратегические и регулярные задания на весь период 3 месяца
- Дальше вступает в игру агент валидации

**SC-07: Валидация агентом (проверка достижимости)**

Агент анализирует:
- Профиль пользователя из @me (его портрет, паттерны)
- Активные цели (общая нагрузка по времени и усталости)
- События календаря (свободные слоты)

Система прикидывает примерный план достижения цели, анализирует доступное время и переходит к SC-08

**SC-08: Генерация квестов (Key Results)**

- Агент генерирует 3-5 квестов на основе описания цели
- Каждый квест содержит:
  - Название (четкое, измеримое)
  - Целевое значение (числовое)
  - Единица измерения (например: "заданий выполнено", "часов практики", "глав прочитано")
  - Начальное значение: 0
- Пользователь может отредактировать квесты (названия, целевые значения) перед подтверждением
- Отправка → генерация заданий (SC-09)

**SC-09: Генерация 3-месячного плана заданий**

Агент генерирует ежедневный план заданий:
- Регулярные задания: расписание интервального повторения (интервалы Эббингауза)
- Стратегические задания: равномерное распределение по 90 дням
- Стоимость усталости назначается каждому заданию
- Система учитывает проекцию усталости при планировании, т.е. запланированные на день задачи не могут превысить 100%
- Все задания планируются в свободные слоты календаря в период активности пользователя
- **Чередование и перерывы (правила расстановки заданий в слоты — см. SC-36)**
- Показ предварительного просмотра плана: календарный вид с количеством заданий на день, проекция усталости
- Подтверждение пользователем → цель становится активной, задания запланированы

**SC-10: Валидация нагрузки (мягкий лимит в 5 целей)**

- НЕТ жесткого лимита на количество активных целей
- При создании новой цели: агент проверяет общую нагрузку, она не дожна превышать 100% по любому из типов активности

**SC-11: Типология целей**

- НЕТ гибридных целей
- Только skill-based (больше регулярных) или knowledge-based (больше стратегических)
- Соотношение гибкое, система решает сама на основе контекста цели

**SC-11b: Страница детали цели (Goal Detail)**

Страница активной цели включает:
- Заголовок: название, сфера, тип, даты
- **Общий прогресс-бар** (Overall Progress): `sum(quest.current) / sum(quest.target) * 100%`, цвет — cyan для skill-целей, purple для knowledge-целей
- **Квесты-аккордеоны** (collapsible): каждый квест показывает прогресс-бар и свёрнут по умолчанию. Раскрытие → список всех задач квеста (title, type badge, дата, статус), отсортированный по scheduled_date
- **Кнопка "Expert Consultation"** → открывает GoalDialogModal (стриминговый чат с AI-экспертом по цели, контекст: цель + квесты + заметки)
- **Кнопка "Goal Notes"** → открывает GoalNotesModal (markdown редактор с автосохранением (debounce 1.5s), содержимое берётся из `notes` таблицы по пути `{sphere}/{goal}/goal.md`)

**SC-12: Переформулировка цели**

- Во время генерации (SC-06 до SC-09): пользователь может итеративно работать с агентом
- Агент может предлагать переформулировки на основе обратной связи валидации
- После создания цели (статус active): НЕЛЬЗЯ переформулировать
- Единственные опции после создания: отменить цель или скорректировать усталость в ретроспективе (SC-27)

### Flow 3: Ежедневное выполнение и отслеживание прогресса

**SC-13: Ночное планирование (логика в 00:00)**

Каждую ночь система автоматически:
- Определяет, какие задания за прошедший день были не выполнены
- Для невыполненных стратегических: запускает алгоритм перераспределения (SC-21)
- Для невыполненных регулярных: увеличивает счетчик пропусков, проверяет условия провала цели (SC-24)
- Сбрасывает усталость на 0/0/0
- Загружает все активные цели пользователя
- Собирает задания, запланированные на новый день (включая регулярные по интервальному графику)
- Расставляет задания в свободные слоты календаря по правилам чередования и перерывов (SC-36)
- Пользователь открывает приложение и видит список заданий на сегодня, сгруппированных по целям, с рекомендованным временем для каждого

**SC-14: Выполнение заданий**

**Регулярные задания:**
- Система предлагает рекомендованное время в календаре, но пользователь может выполнять в любое время дня
- Кнопка "Start" на карточке задания → запускается таймер 12 мин (ActiveTaskBanner появляется вверху всех страниц)
- Завершение (по таймеру или авто) → отметка о выполнении, начисление XP, обновление усталости (следующее повторение уже запланировано по интервальному графику)

**Стратегические задания:**
- Система предлагает рекомендованное время в календаре, но пользователь может выполнять в любое время дня
- Кнопка "Start" → открывается StrategicExecutionDialog (диалог с AI-экспертом) И одновременно запускается таймер 27 мин
- В диалоге: система выступает как эксперт-наставник, помогает разобраться в теме задания
- Для завершения обязательна заметка-отчёт: минимум 50 символов И 8 слов (валидируется и клиентом, и сервером — `note.trim().split(/\s+/).length >= 8`)
- Заметка сохраняется в папку цели в KB
- Завершение → начисление XP, обновление усталости

**Общее:**
- Задание можно выполнить, не выполнить, или начать и отменить (равнозначно невыполнению)
- Нельзя выполнять задания из прошлых/будущих дней — только сегодняшние
- Одновременно активно только одно задание (OneTaskAtATime)
- Невыполненные за день задания обрабатываются ночной логикой планирования (SC-13)

**SC-15: Награды XP и повышение уровня**

- Регулярное задание: 50 XP
- Стратегическое задание: 100 XP
- При выполнении: XP добавляется к общему счету пользователя
- Если XP достигает порога для следующего уровня: повышение уровня
  - Полноэкранный модал в стиле визуальной новеллы
  - Показ: новый уровень, общий XP, требование до следующего уровня
  - Кнопка "Продолжить" → закрытие окна с повышением уровня

**SC-16: Визуализация усталости**

**Верхняя панель пользователя (всегда видна):**
- Аватар пользователя, уровень, полоса XP
- 3 полосы усталости (Физическая, Эмоциональная, Интеллектуальная) с текущим процентом и цветовым кодированием
- Обновляется в реальном времени по мере выполнения заданий
- При наведении: тултип показывает точные значения
- кнопка для перехода на стараницу настроек (там будет выход из профиля, возможность подвязать другой календарь гугл, изменение периода активности и изменение часового пояса и изменение времени ретроспектив)

**SC-17: Предупреждения об усталости**

- Если любой тип усталости достигает 91%+: иконка предупреждения, при наведении текст с рекомендацией отдохнуть
- просто рекомендация, не блокирует выполнение заданий

**SC-18: Ежедневный сброс усталости**

- В 00:00 усталость сбрасывается для всех пользователей (в рамках ночной логики планирования)
- Если у пользователя открыто приложение в полночь — показать уведомление "Новый день начался! Усталость сброшена."

**SC-19: Диалог с системой по цели**

**Роль:** мыслитель, наставник, гуру цели — творческий помощник (в отличие от RAG в KB, который выступает как архивариус и библиотекарь заметок)

**Один сквозной чат на цель:**
- Все сообщения хранятся в БД, пользователь может просматривать всю историю диалога
- Диалог не делится на отдельные чаты — это один непрерывный поток общения по цели

**Контекст диалога (входные переменные, всегда одинаковые):**
- Текущий прогресс по цели (квесты, статистика выполнения)
- Сама цель, квесты, общий план
- Связанные заметки из KB по цели (не все — управление контекстным окном)
- При выполнении стратегического задания: дополнительно подгружается постановка конкретного задания, чтобы система помогала именно с его решением и созданием заметки

**Управление контекстным окном:**
- В LLM не отправляется вся история чата и все заметки — это переполнит контекст
- Система накапливает саммари диалога: при каждом выполнении стратегического задания (и периодически в свободном диалоге) формируется компактное саммари предыдущих обсуждений
- В контекст LLM попадает: саммари + последние N сообщений + входные переменные цели
- Саммари должно быть небольшим по объему, но полезным для решения задачи

**Использование:**
- Основной use case: среда выполнения стратегических заданий (SC-14)
- Дополнительно: пользователь может зайти в диалог по цели в любое время для консультации с системой-экспертом
- Ручное создание заметок на вкладке заметок по цели также возможно (как и через KB)
- Заметки, созданные через диалог или вручную, появляются в дереве файлов KB под соответствующей целью
- Эти заметки используются агентом для анализа ретроспективы и при формировании новой цели при провале

### Flow 4: Адаптация и обработка пропусков

**SC-20: Обнаружение невыполненных заданий**

В рамках ночной логики планирования (00:00):
- Система определяет, какие задания за прошедший день не были выполнены
- Для невыполненных стратегических заданий: запуск перераспределения (SC-21)
- Для невыполненных регулярных заданий: увеличение счетчика пропусков, проверка условий провала (SC-24)

**SC-21: Алгоритм уплотнения расписания стратегических заданий**

**Философия:** Пропущенные задания блокируют последующие (в квесте), поэтому их нужно вместить в ближайшее время. Система "уплотняет" расписание — сдвигает будущие задания, чтобы освободить место для пропущенных.

**Триггеры запуска уплотнения:**
1. Каждую полночь при планировании предстоящего дня (для всех целей с пропущенными заданиями)
2. Каждую полночь для целей с баннером (попытка вместить задания при появлении свободного времени)
3. При отказе от цели (немедленно — освобождаются слоты, пытаемся вместить задания других целей)

**Процесс уплотнения (каждую полночь в 00:00):**

**Шаг 1. Обнаружение пропущенных заданий**
- Система определяет все невыполненные стратегические задания за прошедший день

**Шаг 2. Группировка и приоритизация**
- Группируем пропущенные задания по целям
- Сортируем цели по близости дедлайна (ближайший дедлайн = наивысший приоритет)

 
a) **Формируем очередь пропущенных заданий:**
   - Берём пропущенные задания в порядке их последовательности внутри квеста
   - Задания внутри квеста строго последовательны — нельзя выполнить задание №5 квеста, пока не выполнено №4 этого же квеста

b) **Вмещаем в ближайшие дни:**
   - Вставляем пропущенные задания в ближайшие свободные слоты (начиная с завтрашнего дня)
   - Это сдвигает задания, которые были изначально запланированы на эти дни, на более позднее время
   - Сдвинутые задания занимают следующие свободные слоты, каскадно сдвигая остальные
   - Продолжаем до тех пор, пока все задания не впишутся в дедлайн цели ИЛИ задания выйдут за пределы дедлайна

c) **Результат уплотнения:**
   - **Успех:** все задания вписались в дедлайн → если был баннер "Цель под угрозой", убираем его
   - **Провал:** часть заданий выходит за пределы дедлайна → показываем баннер: "Цель '[Название]' под угрозой: не все задания вмещаются в дедлайн. Вы всё ещё можете достичь цели, если будете выполнять задания внеурочно."

d) **Переход к следующей цели:**
   - Следующая по приоритету цель планируется в оставшиеся свободные слоты (после уплотнения предыдущих целей)
   - Это гарантирует, что самые срочные цели получают слоты первыми и с большей вероятностью не выйдут за пределы дедлайна

**Шаг 4. Проверка целей с баннерами (даже без новых пропусков)**
- Для целей с баннером "Цель под угрозой" повторно запускаем уплотнение
- Возможно, освободилось время (отказ от другой цели, удаление событий календаря, завершение цели)
- Если теперь удаётся вместить все задания → убираем баннер

**Особые случаи:**

**При отказе от цели:**
- Все её задания удаляются → освобождаются слоты в календаре
- Немедленно запускается уплотнение для всех целей с баннерами
- Пытаемся занять освободившееся время заданиями, которые раньше не вмещались

**При удалении событий из Google Calendar:**
- Не делаем ничего в реальном времени
- В полночь при планировании система увидит освободившееся время и выполнит уплотнение для целей с баннерами

**Важные детали:**
- Регулярные задания НЕ участвуют в уплотнении — они идут по фиксированному интервальному графику (алгоритм Эббингауза)
- Правило OKR (70% = 100%): баннер показываем даже при выходе за дедлайн хотя бы одного задания (мотивация пользователя), но технически у пользователя есть запас 30% — даже с баннером цель достижима
- Пользователь может выполнять только задания текущего дня (не может опережать график)

**SC-22: Конфликт с событием календаря**

- Если пользователь добавляет событие в Google Calendar, которое конфликтует со стратегическим или регулярным заданием — в момент добавления ничего не происходит
- Пользователь просто занимается своими делами
- Ночью, при запуске логики планирования, система оценивает, были ли задания реально пропущены за прошедший день
- Если стратегическое задание пропущено — запускается перераспределение (SC-21)
- Если регулярное задание пропущено — увеличивается счетчик пропусков, стандартная логика (SC-24)

**SC-23: Регулярные задания и интервальное повторение**

- Регулярные задания всегда идут по своему интервальному графику (алгоритм Эббингауза)
- Если регулярное задание не выполнено — оно НЕ перепланируется
- Следующее повторение этого задания наступит по графику (через интервал)
- Пропуски регулярных заданий не влияют на их расписание — график фиксированный

**SC-24: Условия провала цели**

Цель считается проваленной, если выполняется любое из условий:
- Одно и то же регулярное задание пропущено 3 раза подряд
- Суммарный процент пропусков одного любого регулярных заданий по цели достигает 20% от общего плана

При провале цели:
- Прогресс по стратегическим заданиям и все заметки к ним сохраняются
- Прогресс по регулярным заданиям сбрасывается
- Система предлагает сформулировать новую цель на основе проваленной, уже с учетом полученных знаний от выполненных стратегических задач
- Счетчик последовательных пропусков конкретного регулярного задания сбрасывается при его успешном выполнении (но общий счетчик пропусков не сбрасывается)

### Flow 5: Ретроспективы, паттерны и обратная связь

**SC-25: Запуск ретроспективы**

**По расписанию:**
- Уведомление в Telegram в запланированное время (из настроек SC-04)
- Баннер в шапке приложения на любой странице с приглашением начать ретроспективу (появляется в запланированное время)
- Начать ретроспективу вручную в любое время нельзя — только по расписанию

**UI:**
- Открывается большой модал (wizard-стиль, пагинированный)
- Нельзя пропускать страницы — только последовательное прохождение
- Навигация: кнопки "Далее" / "Назад" + стрелки клавиатуры
- Следующая страница открывается только после выполнения целевого действия на текущей (просмотр статистики или заполнение формы)

**SC-26: UI ретроспективы (пагинированная модалка-wizard)**

**Структура страниц (последовательно, нельзя пропускать):**

**Страница 1: Общая статистика**
- **Заголовок:** "Еженедельная ретроспектива"
- **Содержание:**
  - Агрегированные данные по всем целям: процент выполнения, заработанный XP
  - Прогресс уровня пользователя
  - График использования усталости (тренды по трём типам)
  - Выявленные паттерны поведения (из @me/patterns.md)
- **Вопрос по усталости (только если есть основания из паттернов):**
  - Если система видит паттерн перегрузки по какому-то типу усталости → показывает вопрос:

**Страницы 2..N: Статистика по каждой цели + форма обратной связи**

Для каждой активной/недавно завершённой цели — отдельная страница:

  - Список невыполненных заданий
  - Паттерны пропусков
- **Форма обратной связи:**
  - **Быстрый вопрос 1:** "Насколько комфортна текущая нагрузка по этой цели?"
    - Варианты: "Легко" / "Нормально" / "Тяжело"
  - **Быстрый вопрос 2 (опционально):** Дополнительный контекстный вопрос от системы на основе паттернов
  - **Текстовое поле (необязательное):** "Что хотите скорректировать в этой цели?"
    - Свободный текст, markdown
- **Действие:** Заполнение формы (минимум выбор в быстрых вопросах) → кнопка "Далее" разблокируется

**Страница N+1: Финальная страница (аппрув изменений)**

- **Заголовок:** "Подтверждение изменений"
- **Содержание:**
  - Лог всех предложенных системой изменений на основе анализа (SC-27):
    - Корректировки усталости (если были одобрены на странице 1)
    - Корректировки содержания заданий по каждой цели (переформулировки, замены задач)
  - Примечание: "Изменения направлены на корректировку содержания задач, а не структуры плана. Количество и распределение заданий остаются примерно теми же."
- **Действие:**
  - Кнопка "Назад" — вернуться на любую предыдущую страницу и изменить ответы
  - Кнопка "Применить изменения" → обновление плана, пересчёт проекций, закрытие модала
  - Нет отката после применения

**SC-27: Анализ агентом и корректировка плана**

**Агент обрабатывает:**
- Статистику выполнения по каждой цели
- Обратную связь пользователя (ответы на быстрые вопросы + текстовые комментарии)
- Паттерны поведения (из @me/patterns.md)
- Заметки из базы знаний по цели (понимание прогресса и сложностей)

**Генерирует:**

**1. Инсайты и новые паттерны:**
- Выявленные паттерны (например: "Вы пропускаете задания с высокой эмоциональной нагрузкой по понедельникам")
- Новые паттерны добавляются в @me/patterns.md

**2. Корректировки усталости (если одобрено на странице 1):**
- Снижение стоимости усталости для заданий определённого типа
- Можно только уменьшать (не увеличивать)

**3. Корректировки содержания заданий (основной фокус):**
- **Переформулировка заданий:** более точные формулировки на основе обратной связи
- **Замена заданий:** если задание оказалось нерелевантным или слишком сложным
- **Уточнение формулировок квестов:** если цель или KR были неясны

**Важно:**
- План по нагрузке НЕ меняется значительно: количество заданий, их распределение по дням остаются примерно теми же
- Фокус на улучшении СОДЕРЖАНИЯ: что именно делать, а не когда и сколько
- Цель НЕ должна стать недостижимой после ретроспективы

**Финальная страница (аппрув):**
- Пользователь видит лог всех предложенных изменений
- Может вернуться назад на любую страницу и скорректировать ответы
- Кнопка "Применить изменения" → обновление расписания, пересчёт проекций
- Нет отката после применения

**Хранение:**
- В `@me/patterns.md` как markdown список с датами и описаниями паттернов

**Агент использует паттерны для:**

### Flow 6: База знаний (Obsidian-совместимая)

**SC-29: Инициализация базы знаний**

**При первом запуске (во время онбординга):**

- Генерация эмбеддингов для @me файлов (начальные данные профиля)

**При создании сферы:**
- Создание файла сферы в базе знаний

**При создании цели:**
- Создание файла цели с метаданными в базе знаний

**SC-30: Создание и редактирование заметок**

**Создание заметок (из левой панели - дерева файлов):**
- Правый клик на папке (сфера/цель/@me) → "Создать заметку"
- Или кнопка "+" рядом с названием папки → "Создать заметку"
- Вводится название заметки → создаётся файл в выбранной папке
- Автоматическое создание при добавлении заметки к стратегическому заданию (SC-19) — заметка создаётся в папке цели

**Управление файлами (из левой панели):**
- **Переименование:** правый клик на файле → "Переименовать"
- **Удаление:** правый клик на файле → "Удалить" (с подтверждением)
- **Перемещение:** drag-and-drop файла между папками
- Всё управление файлами происходит из дерева директорий слева (как в Obsidian)

**Редактирование (в центральной панели):**
- Клик на заметку в дереве слева → открывается в центральной панели
- Кнопка "Редактировать" → переход в режим редактирования
- Markdown редактор с подсветкой синтаксиса
- Автодополнение для `[[wikilinks]]` (из существующих заметок)
- Автодополнение для `#тегов` (из существующих тегов)
- Загрузка изображений: drag-drop → вставка `![[имя-файла.png]]`
- Автосохранение
- При сохранении: извлечение wikilinks/тегов, обновление метаданных, регенерация эмбеддингов

**SC-31: Просмотр заметки (три панели)**

**Левая панель:**
- Дерево файлов: @me → сферы → цели → заметки
- Строка поиска: фильтрация по названию, контенту
- Список тегов: клик по тегу → фильтрация заметок
- Переключатель вида графа

**Центральная панель:**
- Отрендеренный markdown в стиле дизайн-системы
- Wikilinks: кликабельны, при наведении показывают превью
- Теги: кликабельны, фильтруют левую панель
- Секция backlinks внизу
- Кнопка редактирования

**Правая панель:**
- Интерфейс RAG чата
- Контекст: текущая заметка + связанные
- История чата (только для сессии, не сохраняется)
- Предлагаемые промпты

**SC-32: RAG чат в правой панели**

**Сборка контекста:**
- Текущая заметка (полный контент)
- Связанные заметки через wikilinks (2 уровня вглубь)
- Семантический поиск по всей базе знаний

**Процесс чата:**
- Пользователь вводит вопрос → агент обрабатывает с контекстом RAG
- Ответ включает markdown с ссылками на источники (кликабельны, переход к заметке)
- Агент может предложить создать новую заметку или добавить wikilink

**SC-33: Управление связями и граф**

**Автоматические связи:**
- Wikilinks создают двунаправленные связи
- Панель backlinks: показывает все заметки, ссылающиеся на текущую

**Семантические связи (AI предложения):**
- После сохранения заметки: агент анализирует контент, предлагает связанные заметки
- Показывается ненавязчиво, можно закрыть

**Вид графа:**
- Визуализация связей между заметками
- Узлы разного размера и цвета в зависимости от типа и количества связей
- Клик по узлу → переход к заметке
- Фильтры: по сфере, по цели, по тегу
- Зум, панорамирование

**SC-34: Файлы @me и обновления профиля**

- @me файлы редактируются как обычные заметки
- Frontmatter для структурированных данных, тело для свободного текста
- При изменении @me файлов: регенерация эмбеддингов, пересчет валидации для активных целей (если изменился рабочий период)
- patterns.md: read-only UI, показывает паттерны от агента (SC-28)

### Flow 7: Интеграция с Google Calendar

**SC-35: OAuth и настройка синхронизации**

- Подключение обязательно при онбординге (SC-01, шаг 3)
- OAuth 2.0 flow с Google: скоуп `calendar.events` (чтение + создание/управление событиями)
- После авторизации: токен сохраняется зашифрованным (AES-256-GCM) в поле `users.calendar_token_encrypted`; refresh token используется для автообновления
- Периодическое обновление событий + push-уведомления от Google при изменениях в календаре

**SC-36: Планирование заданий в свободные слоты**

- При подтверждении цели (SC-09): задания немедленно синхронизируются с Google Calendar (fire-and-forget, после ответа). Каждое задание получает поле `calendar_event_id`
- При отмене цели (cancel) или провале (fail): все связанные события удаляются из Google Calendar (fire-and-forget)
- При ночном планировании (SC-13): задания без `calendar_event_id` синхронизируются через `/api/calendar/sync-tasks`
- Все задания (и регулярные, и стратегические) планируются в свободные слоты периода активности пользователя
- Для каждого задания система предлагает рекомендованное время на основе паттернов пользователя (пока нет паттернов — на основе научных рекомендаций ниже)
- Если свободных слотов не хватает для всех заданий — алгоритм приоритизации (стратегические приоритетнее)

**Правила чередования заданий (interleaving):**
- Не ставить подряд задания с одинаковым доминирующим типом усталости (например, два подряд интеллектуально нагруженных задания). Система чередует: интеллектуальное → физическое → эмоциональное
- Чередовать задания из разных целей/квестов, а не блоками по одной цели. Interleaving effect: чередование даёт лучшее долгосрочное запоминание, чем блочная практика
- Исключение: в самом начале цели (первые 1-2 недели) допускается блочное размещение заданий одного квеста для формирования базы — после чего переход к чередованию

**Правила перерывов между заданиями:**
- После регулярного задания (10-15 мин): перерыв 5 мин
- После стратегического задания (25-30 мин): перерыв 10 мин
- После каждых ~90 мин суммарной работы (или 4 заданий подряд, что наступит раньше): длинный перерыв 15-20 мин (ультрадианный ритм)
- Перерывы — рекомендация, а не блокировка: если пользователь хочет продолжить раньше, система не запрещает
- В расписании перерывы отображаются как свободные промежутки между запланированными заданиями (не как отдельные события)

**SC-37: Визуализация расписания в дневном виде**

- В дневном виде видны и задания, и события Google Calendar
- Все задания имеют рекомендованное время, отображаются в соответствующих слотах
- События Google Calendar отображаются как занятые слоты
- Пользователь может выполнять задания в любое время, рекомендованное время носит информационный характер

**SC-38: Реакция на изменения в календаре**

- При добавлении/изменении/удалении событий в Google Calendar — в реальном времени ничего не предпринимается
- Ночью, при запуске логики планирования (SC-13), система оценивает фактические результаты прошедшего дня
- Если задания были пропущены (по любой причине, включая конфликт с новым событием) — стандартная логика: перераспределение для стратегических (SC-21), счетчик пропусков для регулярных (SC-24)
- При отключении календаря: задания остаются, но система не может планировать их в свободные слоты

## 5. Критические бизнес-правила (краткая справка)

### Правила целей:
- Всегда 3 месяца (90 дней), без продлений, без кастомных дедлайнов
- При создании: пользователь формулирует цель свободно, система оценивает достижимость → если нереально, предлагает упрощённый вариант
- Полный план генерируется сразу на все 90 дней (пользователь видит весь план наперёд)
- Нет переформулировки после создания (только во время генерации)
- Нет жесткого лимита активных целей (естественные ограничения — время и усталость)
- Skill-based: больше регулярных, меньше стратегических (гибкое соотношение)
- Knowledge-based: больше стратегических, меньше регулярных (гибкое соотношение)
- 3-5 квестов на цель, квесты выполняются параллельно
- Задания внутри квеста — строго последовательны (нельзя сделать №5 без №4)
- Задания разных квестов — независимы

### Правила заданий:
- Регулярные: 12 мин (`duration_minutes=12`), 50 XP, интервальное повторение (1, 2, 4, 7, 14, 30, 60 дней), не перепланируются
- Стратегические: 27 мин (`duration_minutes=27`), 100 XP, через AI-диалог (обязательная заметка ≥50 символов И ≥8 слов)
- Все задания планируются в свободные слоты календаря с рекомендованным временем (полный план на 90 дней создаётся сразу)
- Каждую полночь: планирование завтрашнего дня с учётом пропусков (уплотнение расписания)
- Пользователь может выполнять в любое время дня, но только сегодняшние задания
- Нельзя выполнять задания из прошлых/будущих дней
- Старт задания — кнопка "Start" → таймер (ActiveTaskBanner). Одновременно одно задание. При 0 — автозавершение

### Условия провала цели:
- 3 последовательных пропуска КОНКРЕТНОГО регулярного задания → провал цели
- 20% суммарных пропуско�� конкретного регулярного заданий по цели → провал цели
- При провале: стратегический прогресс и заметки сохраняются, регулярный прогресс сбрасывается
- Система предлагает создать новую цель на основе проваленной

### Правила усталости:
- 3 типа: Физическая, Эмоциональная, Интеллектуальная (0-100%)
- Стоимость: 4% за регулярное задание, 6% за стратегическое
- Дневной лимит: не более 100% по любому типу, но при этом система только предупреждает но не запрещает делать задания при полной устолости
- Сброс: 00:00 ежедневно
- Визуализация: всегда в верхней панели пользователя

### Правила расписания (чередование и перерывы):
- Не ставить подряд задания с одинаковым доминирующим типом усталости — чередовать типы
- Чередовать задания из разных целей (interleaving effect), кроме первых 1-2 недель цели
- Перерыв после регулярного: 5 мин; после стратегического: 10 мин
- Длинный перерыв 15-20 мин после ~90 мин работы или 4 заданий подряд
- Перерывы — рекомендация, не блокировка

### Правила уплотнения (перераспределения):
- Только для стратегических заданий
- Каждую полночь: пропуски → группировка по целям → сортировка по дедлайну
- Уплотнение: вставка пропущенных в ближайшие слоты → сдвиг будущих → каскад
- Если не помещается → баннер + раздел "Незапланированные"
- Триггеры: пропуски, отказ от цели, удаление событий календаря

### Правила ретроспективы:
- Wizard-стиль: страницы с "Далее"/"Назад", нельзя пропускать
- Страница 1: общая статистика + вопрос по усталости
- Страницы 2..N: статистика цели + форма обратной связи
- Финальная страница: лог изменений + аппрув
- Фокус на изменении СОДЕРЖАНИЯ задач, не структуры плана
- Паттерны хранятся в @me/patterns.md

### Правила базы знаний:
- Obsidian-совместимый формат (markdown, wikilinks, тег��)
- Все заметки привязаны к целям (не к отдельным заданиям)
- Заметки стратегических заданий автоматически создаются в папке цели
- @me/patterns.md: read-only, обновляется агентом

### Правила календаря:
- Подключение обязательно при онбординге; скоуп `calendar.events` (чтение + запись событий)
- Все задания планируются в свободные слоты календаря
- При подтверждении цели: все задания сразу создаются как события Google Calendar (fire-and-forget)
- При отмене или провале цели: все события цели удаляются из Google Calendar (fire-and-forget)
- Конфликты обрабатываются ночью при планировании (не в реальном времени)
- При отключении календаря система не может планировать задания в слоты

## 8. Обоснование технических решений

### Почему Vercel AI SDK (вместо LangGraph.js)?
- Нативная интеграция с Next.js (streaming из API routes)
- `streamText` + `generateObject` — структурированный вывод из коробки
- Tool use без лишних абстракций
- Проще в поддержке: меньше зависимостей, меньше moving parts

### Почему Anthropic SDK напрямую (вместо OpenRouter)?
- Прямой доступ к последним моделям без прокси-задержек
- Модели подобраны точно под задачи: Haiku 4.5 для рутинных агентов (скорость/цена), Sonnet 4.6 для сложных (качество)
- Официальная поддержка streaming и tool use

### Почему Obsidian-совместимый формат?
- Проверенный markdown стандарт (wikilinks, теги)
- Пользователь может экспортировать/импортировать в Obsidian desktop
- Меньше разработки кастомного формата (фокус на UI и RAG)

### Почему Supabase?
- PostgreSQL с pgvector (эмбеддинги для RAG)
- Realtime подписки
- Storage только для изображений (контент заметок — в PostgreSQL TEXT)
- Встроенная аутентификация
